2.H5 web storage

localStroage 和 sessionStorage

localStorage 是 HTML5 标准中新加入的技术，它并不是什么划时代的新东西。 早在 IE 6 时代，就有一 个叫 userData 的东西用于本地存储，而当时考虑到浏览器兼容性， 更通用的方案是使用 Flash。而如今，localStorage 被大多数浏览器所支持，更多用他来存储数据。

sessionStorage 与 localStorage 的接口类似， 但保存数据的生命周期与 localStorage 不同。直译过来是“会话”。 而 sessionStorage 是一个前端的概念， 它只 是可以将一部分数据在当前会话中保存下来， 刷新页面数据依旧存在。 但当页面关闭后，sessionStorage 中的数据就会被清空

3.Cookie

Cookie（也叫Web cookie或者浏览器Cookie）是服务器发送到用户浏览器并保存在浏览器上的一块数据， 它会在浏览器下一次发起请求时被携带并发送到服务器上。比较经典的，可以它用来确定两次请求是否来自于同一个浏 览器，从而能够确认和保持用户的登录状态。Cookie的使用使得基于无状态的HTTP协议上记录稳定的状态信息成为了可能。

几种方式孰优孰劣？

URL传参

优势：

1.简洁易用，可同时传递多个字符型参数

2.可以很方便的在多个页面之间传递参数，无需额外处理，对性能影响微乎其微

劣势：

1.安全性没有保障，传递的参数直接显示在URL地址中

2.长度受限，最大为2K

3.只能传递字符型参数，由于编码原因，传递中文时容易出错，显示乱码或解析错误等

H5 web storage

优势：

1.sessionStorage在页面关闭后删除，localStorage永久储存，除非手动删除。分别适用于不同情况。

2.可以存储最大5M的数据，保存在浏览器中用于性能优化，可以提高页面的浏览速度

劣势：

兼容性问题，h5 web storage要求浏览器至少要支持到4MB，对于当今主流浏览器可以使用，但ie8之前的版本不适用。

cookie

优势：

1.可以永久保存，同时可以设置存活周期。

2.记录用户的信息，例如登录信息，购物车等，让用户二次访问页面时不用重复操作

劣势：

1.每次网络请求 Request headers 中都会带上Cookie。所以如果 Cookie太多太大对传输效率会有影响。

2.一般浏览器存储Cookie最大容量为4k，所以大量数据不要存到Cookie。

8.更多讨论

用哪种传参方式比较好

答：这个就是活学活用，根据不同情况来使用不同的方法

比如记录登录信息用cookie，页面内表格的翻页可以用URL控制，大型的文件用H5

为什js本地网页无法设置cookie？

答：出于安全考虑，chrome本地环境cookie是禁止的，可以给他个服务器环境，或者用firebox。

传参是按值传递还是引用传递

答：js中所有的参数都是传值类型，当我们将一个值传递到函数内部时，一个临时的局部变量会被创建，形成对这个参数的一个拷贝，任何对该值的改变都不会影响原有的外部变量

语法： return[()[expression][]]; 可选项 expression 参数是要从函数返回的值。如果省略，则该函数不返回值。 用 return 语句来终止一个函数的执行，并返回 expression 的值。如果 expression 被省略， 或在函数内没有 return 语句被执行，则把值 undefined 赋给调用当前函数的表达式。
